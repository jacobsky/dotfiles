# Writing Style Guidelines

## Overview

This style guide establishes writing standards for OpenCode technical documentation to ensure consistency, clarity, and professional quality across all documentation types.

## Voice and Tone

### General Principles

#### Clear and Direct
- Use clear, straightforward language
- Avoid unnecessary complexity and jargon
- Get to the point quickly
- Focus on what users need to know

#### Professional and Approachable
- Maintain professional tone without being overly formal
- Use contractions sparingly for readability
- Address the user directly when appropriate
- Maintain confidence and authority

#### Helpful and Encouraging
- Assume users are intelligent but busy
- Provide context and rationale
- Encourage learning and exploration
- Acknowledge complexity without overwhelming

### Specific Guidelines

#### Voice Characteristics
- **Active Voice**: Prefer active over passive voice
- **Present Tense**: Use present tense for current functionality
- **Second Person**: Use "you" when addressing the user directly
- **Consistent Perspective**: Maintain consistent viewpoint throughout

#### Tone Variations by Document Type
- **Tutorials**: Encouraging, step-by-step, supportive
- **Reference Docs**: Concise, factual, authoritative
- **Conceptual Guides**: Explanatory, contextual, educational
- **API Documentation**: Precise, technical, complete

## Language and Terminology

### Technical Language

#### Define Technical Terms
- Define technical terms on first use
- Provide brief explanations for complex concepts
- Use analogies when helpful for understanding
- Include glossaries for extensive terminology

#### Jargon Management
- Use standard industry terminology
- Avoid unnecessary jargon and buzzwords
- Explain domain-specific terms
- Maintain consistency in term usage

#### Acronyms and Abbreviations
- Define acronyms on first occurrence
- Use standard abbreviations consistently
- Avoid creating new abbreviations
- Spell out terms frequently for clarity

### Word Choice

#### Precision and Accuracy
- Use precise technical language
- Avoid ambiguous terms
- Choose words with clear meanings
- Maintain technical accuracy

#### Simplicity and Clarity
- Prefer simple words over complex ones
- Use familiar terms when possible
- Break down complex concepts
- Avoid unnecessary modifiers

#### Consistency
- Use the same term for the same concept
- Maintain consistent naming conventions
- Follow established terminology patterns
- Create term dictionaries for projects

## Sentence Structure and Style

### Sentence Guidelines

#### Length and Complexity
- Aim for 15-25 words per sentence on average
- Break long sentences into shorter ones
- Vary sentence length for readability
- Avoid complex nested structures

#### Structure Patterns
- Start sentences with clear subjects
- Use parallel structure for lists
- Maintain logical flow between sentences
- Use transition words for clarity

#### Common Structures to Avoid
- Excessive subordinate clauses
- Long introductory phrases
- Multiple negatives
- Overly complex conditional statements

### Paragraph Structure

#### Organization
- Focus each paragraph on a single idea
- Start with topic sentences
- Provide supporting details
- Use transition sentences between paragraphs

#### Length Guidelines
- Keep paragraphs to 3-6 sentences when possible
- Break up very long paragraphs
- Use bullet points for lists of items
- Employ white space effectively

#### Logical Flow
- Arrange ideas logically
- Use chronological order for processes
- Group related information together
- Maintain consistent organizational patterns

## Formatting and Presentation

### Headings and Structure

#### Heading Hierarchy
- Use proper heading hierarchy (H1 → H2 → H3)
- Don't skip heading levels
- Keep headings descriptive and concise
- Use sentence case for headings (except H1)

#### Heading Content
- Make headings informative
- Use action-oriented headings for guides
- Include keywords for searchability
- Maintain consistent heading styles

#### Section Organization
- Group related content under clear headings
- Use subheadings for complex topics
- Provide introductions for major sections
- Include summaries when helpful

### Lists and Bullet Points

#### When to Use Lists
- For sequential steps or procedures
- For items of equal importance
- For options or alternatives
- For requirements or prerequisites

#### List Formatting
- Use parallel structure for list items
- Start each item with consistent capitalization
- Use appropriate punctuation (periods for complete sentences)
- Limit list length for readability

#### Types of Lists
- **Numbered lists**: For ordered steps
- **Bulleted lists**: For unordered items
- **Definition lists**: For term explanations
- **Nested lists**: For hierarchical information

### Code and Technical Elements

#### Code Blocks
- Use proper syntax highlighting
- Include complete, working examples
- Show context and imports
- Add explanatory comments when needed

#### Inline Code
- Use backticks for code elements
- Format function names and variables consistently
- Include file paths and commands in code format
- Use monospace font for technical terms

#### Technical Formatting
- Use consistent terminology
- Format configuration values clearly
- Show command-line examples accurately
- Include version information when relevant

## Visual Elements

### Images and Diagrams

#### When to Use Visuals
- Complex concepts that benefit from visualization
- Architecture or system overviews
- User interface demonstrations
- Process flows and relationships

#### Image Guidelines
- Use high-quality, clear images
- Ensure appropriate resolution
- Include descriptive alt text
- Maintain consistent visual style

#### Diagram Standards
- Use standard diagramming conventions
- Keep diagrams simple and clear
- Include legends and labels
- Use consistent colors and symbols

### Tables and Data Presentation

#### Table Usage
- For comparing multiple items
- For structured data presentation
- For configuration options
- For quick reference information

#### Table Formatting
- Use clear, descriptive headers
- Align columns consistently
- Include units where applicable
- Use appropriate table complexity

## Accessibility and Inclusion

### Language Accessibility

#### Clear Communication
- Use plain language when possible
- Define complex terms and acronyms
- Provide context for technical concepts
- Consider different expertise levels

#### Cultural Sensitivity
- Avoid culturally specific references
- Use inclusive language
- Consider time zone and date formats
- Be mindful of measurement units

### Visual Accessibility

#### Color and Contrast
- Ensure sufficient contrast ratios
- Don't rely on color alone for meaning
- Use patterns and textures in addition to color
- Consider colorblind users

#### Structure and Navigation
- Use proper heading structure
- Provide alternative text for images
- Ensure keyboard navigation
- Use descriptive link text

## Documentation Types and Styles

### Tutorials

#### Characteristics
- Step-by-step instructions
- Learning-oriented approach
- Practical examples and exercises
- Clear goals and outcomes

#### Writing Style
- Use imperative mood for instructions
- Include explanation and context
- Provide troubleshooting tips
- Encourage experimentation

### Reference Documentation

#### Characteristics
- Comprehensive and complete
- Quick lookup format
- Technical precision
- Consistent structure

#### Writing Style
- Use factual, objective tone
- Include all parameters and options
- Provide syntax specifications
- Use structured format consistently

### Conceptual Guides

#### Characteristics
- In-depth explanations
- Context and background
- Relationships between concepts
- Design rationale

#### Writing Style
- Use explanatory, educational tone
- Provide context and motivation
- Include analogies and examples
- Address why and how questions

### API Documentation

#### Characteristics
- Technical accuracy
- Complete specifications
- Example usage
- Error handling

#### Writing Style
- Use precise technical language
- Include all endpoints and parameters
- Provide request/response examples
- Document error codes and handling

## Quality Standards

### Accuracy and Completeness

#### Technical Accuracy
- Verify all technical information
- Test all code examples
- Check version compatibility
- Review with technical experts

#### Completeness
- Cover all features and options
- Include edge cases and limitations
- Provide error handling information
- Address common questions

### Consistency and Standards

#### Style Consistency
- Follow established style guidelines
- Use consistent terminology
- Maintain uniform formatting
- Apply patterns consistently

#### Standard Compliance
- Follow industry documentation standards
- Adhere to OpenCode skill requirements
- Meet accessibility guidelines
- Comply with legal requirements

- Peer review process

#### Quality Assurance
- Automated validation tools
- Manual checklist review
- User feedback incorporation
- Regular update schedule

### Maintenance Standards

#### Update Processes
- Regular content review schedule
- Version control integration
- Change documentation
- Archive historical versions

#### Continuous Improvement
- Collect user feedback

- Update best practices
- Refine guidelines based on experience

## Common Pitfalls to Avoid

### Language Issues

#### What to Avoid
- Overly complex sentence structures
- Excessive technical jargon
- Inconsistent terminology
- Passive voice overuse

#### Best Practices
- Use clear, direct language
- Define technical terms
- Maintain consistency
- Prefer active voice

### Structure Problems

#### What to Avoid
- Illogical organization
- Missing navigation elements
- Inconsistent formatting
- Poor section hierarchy

#### Best Practices
- Plan content structure carefully
- Include clear navigation
- Apply formatting consistently
- Use proper heading hierarchy

### Content Issues

#### What to Avoid
- Incomplete information
- Outdated content
- Non-functional examples
- Missing context

#### Best Practices
- Ensure comprehensive coverage
- Keep content current
- Test all examples
- Provide appropriate context
